# Cross-alias session visibility — live verification evidence
# Captured: 2026-06-20  (claude_toolkit, branch main)

## Goal
Prove that sessions created under ANY alias (claudeN, deepseek, opencode, xiaomi, …)
are visible from every other alias, and that /resume can continue any session from
any alias. Memory, project settings, and session data are fully shared.

## 1. Mechanism
- `claude-sync-state.sh` now includes provider dirs in its merge targets (not just
  account dirs). Provider dirs (~/.claude-prov-*) are discovered alongside account
  dirs (~/.claude-*).
- `cma_run_provider` wrapper now calls `claude-sync-state pull` before launch and
  `claude-sync-state push` after exit — same pattern as `cma_run` for claudeN aliases.
- `cma_merge_claude_json` merges the `projects` map (including `lastSessionId`,
  `allowedTools`, MCP config, etc.) across ALL dirs, preserving per-dir auth keys.
- The `sessions/` directory is already shared via symlink (CMA_SHARED_ITEMS), so
  session files are inherently visible from all aliases.

## 4. How /resume works across aliases
When you run `claude --resume` (or use /resume in the TUI):
1. Claude Code reads the `sessions/` directory for available sessions
2. It reads `.claude.json` projects map for `lastSessionId` per project
3. Both are now synced across ALL aliases before/after every session

So:
- Start a session as `deepseek` in /path/to/project
- Exit the session (sync-state push runs automatically)
- Run `claude2` in the same /path/to/project
- `/resume` will show the deepseek session (because sessions/ is shared AND
  .claude.json lastSessionId was merged)

## 6. Account-detection invariant preserved
Provider dirs are STILL excluded from account detection (cma_detect_accounts).
They are ONLY included in sync-state's extended detection (claude-sync-state.sh).
This means claude-unify and claude-add-account are unaffected.

## Honest notes
- The sync-state pull/push adds ~1-2 seconds of overhead per launch (jq merge of
  .claude.json across all dirs). This is the same overhead that claudeN aliases
  already have.
- If a session is in-progress in one alias and you try to resume it from another,
  the session file may be locked. This is a Claude Code limitation, not ours.
- The `sessions/` directory was already shared via symlink — the new work ensures
  `.claude.json` project settings (including `lastSessionId`) are also merged.
